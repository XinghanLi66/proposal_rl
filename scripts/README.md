# Ablation Scripts

Eight ablation experiments that systematically vary **prompt strategy**, **fine-tuning mode**, and **reward signal** on top of the SFT checkpoint.
Each script runs a 2×2 hparam search (LR × KL coefficient) and then full training with the best combo.

---

## Quick start

```bash
# Run one experiment (foreground, log to /tmp/expXX.log)
bash scripts/ablations/exp01_baseline.sh 2>&1 | tee /tmp/exp01.log

# Or in a dedicated tmux window
tmux new-window -n exp01
tmux send-keys -t exp01 "bash scripts/ablations/exp01_baseline.sh 2>&1 | tee /tmp/exp01.log" Enter
```

Environment variables you can override:

| Variable        | Default | Purpose                         |
|-----------------|---------|---------------------------------|
| `NGPU`          | `8`     | GPUs per node                   |
| `DASHBOARD_PORT`| `8080`  | Port where `dashboard/server.py` listens |

## Experiment table

| Script | Label | Prompt strategy | Finetune | Reward | Key question |
|--------|-------|-----------------|----------|--------|--------------|
| `exp01_baseline.sh` | baseline | `top_k_refs` | full | embed-PRS | Reference point for all other comparisons |
| `exp02_full_refs.sh` | full-refs | `full_refs` | full | embed-PRS | Does giving *all* refs (truncated) beat top-K LLM selection? |
| `exp03_related_work.sh` | related-work | `related_work` | full | embed-PRS | Does an LLM-synthesized related-work paragraph beat a raw list? |
| `exp04_topk_related_work.sh` | top-k+related-work | `top_k_related_work` | full | embed-PRS | Two-stage LLM: first select top-K, then narrate. Best of both? |
| `exp05_topk_research_question.sh` | top-k+RQ | `with_research_question` | full | embed-PRS | Does appending an LLM-generated research question help? |
| `exp06_lora.sh` | lora | `top_k_refs` | LoRA | embed-PRS | LoRA vs. full fine-tuning (same prompt/reward as baseline) |
| `exp07_fas.sh` | embed-FAS | `top_k_refs` | full | embed-FAS | FAS (future-alignment score) vs. PRS as the training signal |
| `exp08_llm_prs.sh` | llm-judge-PRS | `top_k_refs` | full | LLM-judge via FAS path | LLM judge instead of embedding similarity for the reward |

### Prompt strategies explained

- **`top_k_refs`** — LLM picks the K most relevant references from the list, then those refs are fed as a numbered block to the proposal generator.  
- **`full_refs`** — All references (up to `max_refs=40`, each truncated to 400 chars) are included verbatim. No LLM selection step.  
- **`related_work`** — LLM synthesises a short related-work paragraph from all refs; that paragraph replaces the reference list.  
- **`top_k_related_work`** — Two-stage: LLM first selects top-K refs, then synthesises a related-work paragraph from those K refs only.  
- **`with_research_question`** — Same as `top_k_refs` but the prompt also includes a 1-3 sentence research question generated by the LLM.

## Output layout

Each run creates its own directory under `runs/exps/`:

```
runs/exps/<EXP_NAME>_<TIMESTAMP>/
├── config_base.yaml             # base config (strategy/reward/mode baked in)
├── config_mini_lr*_kl*.yaml     # per hparam-combo mini configs
├── config_full.yaml             # full-training config with best hparams
├── hparam_search.json           # all 4 mini results + best combo
├── .best_hparams                # sourced by _lib.sh: BEST_LR, BEST_KL
├── logs/
│   ├── rl_<EXP_ID>_mini_lr*_kl*.log   # mini-run training logs
│   └── rl_<EXP_ID>.log                # full-training log
└── rl/
    ├── checkpoint-*/            # periodic checkpoints
    └── final/                   # final merged weights
```

## Running 2 experiments per machine (4-machine setup)

Each script allocates a random free port for `torchrun`, so two scripts running simultaneously on the same machine will not collide on the master port.

**Suggested allocation across 4 machines** (each machine gets 2 experiments):

| Machine | Tmux A | Tmux B |
|---------|--------|--------|
| machine-1 | exp01 (baseline) | exp02 (full-refs) |
| machine-2 | exp03 (related-work) | exp04 (top-k+related-work) |
| machine-3 | exp05 (top-k+RQ) | exp06 (lora) |
| machine-4 | exp07 (embed-FAS) | exp08 (llm-judge-PRS) |

Start both on one machine like this:

```bash
# window A
tmux new-session -d -s expA
tmux send-keys -t expA "bash scripts/ablations/exp01_baseline.sh 2>&1 | tee /tmp/exp01.log" Enter

# window B (separate window — both share all 8 GPUs, so schedule on different machines!)
tmux new-session -d -s expB
tmux send-keys -t expB "bash scripts/ablations/exp02_full_refs.sh 2>&1 | tee /tmp/exp02.log" Enter
```

> **GPU contention note**: each experiment uses all `NGPU` GPUs. Two concurrent experiments on the same node will compete for the same 8 GPUs and be very slow. The table above assumes each machine is dedicated to one experiment pair — set `NGPU=4` if you genuinely want to share a node.

## Monitoring

### Dashboard (web UI)

Each script self-registers with the training dashboard as soon as it starts.  
Start the server once on any machine that can reach the `runs/` directory:

```bash
conda run -n loongflow_ml python dashboard/server.py --port 8080
# Open http://<host>:8080 — each experiment appears as its own tab
```

### Log-based monitoring from any machine

```bash
# Tail the hparam search progress
tail -f /tmp/exp01.log | grep -E '\[exp|Mini-run|mean_reward|Best hparam'

# Tail the full training reward curve
tail -f /newcpfs/lxh/agentic-training/proposal_rl/runs/exps/<EXP_ID>/logs/rl_<EXP_ID>.log \
  | grep -E 'reward|loss|step'

# Watch for errors
tail -f /tmp/exp01.log | grep -iE 'error|warning|traceback|EADDR'
```

### Checking all experiments at once

```bash
# Show last reward from every active full-training log
for f in /newcpfs/lxh/agentic-training/proposal_rl/runs/exps/*/logs/rl_exp*.log; do
    last=$(grep -o "'reward': [0-9.e-]*" "$f" | tail -1)
    echo "$(basename "$(dirname "$(dirname "$f")")"): $last"
done
```

## Prerequisites

- **exp07** and **exp08** require a pre-built val-set embedding index:
  ```bash
  conda run -n loongflow_ml python eval/build_index.py --split val
  # produces: runs/eval/val_index.npz
  ```
- All other experiments only need the SFT checkpoint at `runs/sft/final/` and the training dataset at `runs/dataset/train.jsonl`.

## Shared library (`_lib.sh`)

All 8 scripts `source scripts/ablations/_lib.sh` which provides:

| Function | Purpose |
|----------|---------|
| `setup_experiment` | Creates `EXP_DIR`, builds base config via `scripts/make_config.py` |
| `register_dashboard` | POSTs metadata to `http://localhost:$DASHBOARD_PORT/api/experiments` |
| `run_hparam_search` | Runs 2×2 mini-grid (512 examples, 1 epoch each), picks best combo |
| `run_full_training` | Runs full training with `BEST_LR` / `BEST_KL` |
| `start_dashboard_refresh` | Background loop that pings dashboard every 30 s to keep status fresh |
